Bookkeeping System Connection Map

Boosup is a Next.js app backed by Supabase PostgreSQL. Pages live in app/, UI blocks live in components/, reusable calculations live in lib/, and the database structure lives in supabase/.

Sales Flow

How creating or editing a sales bill connects through the system.

  1. app/sales/create/page.tsxLoads products, customers, company settings, and existing sale data when editing.
  2. components/sales/SalesForm.tsxCollects bill number, customer, date, items, tax, discount, payment, and notes.
  3. app/actions.ts: upsertSaleValidates input, calculates totals, writes sales, sales_items, optional sales_payments, and activity_logs.
  4. lib/financial.tsProvides calculateSalesPaymentState for received amount, status, and remaining balance safety.
  5. app/sales/page.tsx and customer profileRead saved sales and payment history for reports and ledgers.

Customer Payment Flow

This is the newest safer transactional flow.

  1. app/customers/[id]/page.tsxShows customer ledger, linked invoices, grouped payment receipts, and invoice payment history.
  2. CustomerPaymentModalCollects payment amount, method, date, and note.
  3. app/actions.ts: createCustomerPaymentValidates basic form values and calls the Supabase RPC.
  4. record_customer_payment_with_allocations RPCIn one PostgreSQL transaction, inserts customer_payments, inserts allocations, updates invoice paid/status, and inserts sales_payments.
  5. Tablescustomer_payments -> customer_payment_allocations -> sales -> sales_payments.
If any invoice allocation or payment-history insert fails, the whole customer payment rolls back.

Purchase Flow

Purchase bill creation and payment history.

  1. app/purchases/create/page.tsxLoads vendors and existing purchase data when editing.
  2. PurchaseFormCollects supplier, items, payment status, amount paid now, and notes.
  3. app/actions.ts: upsertPurchaseWrites purchases, purchase_items, optional purchase_payments, and activity_logs.
  4. lib/financial.tsProvides calculatePurchasePaymentState and resolvePurchasePaymentStatus.
  5. app/purchases/page.tsxDisplays purchases, expenses, and recent payments.

Supplier Payment Flow

Supplier-level payment allocation to oldest unpaid purchase bills.

  1. app/vendors/[id]/page.tsxShows supplier profile, bills, payable balances, supplier payments, and bill payment history.
  2. SupplierPaymentModalCollects supplier payment amount, method, date, and note.
  3. app/actions.ts: createSupplierPaymentAllocates payment FIFO, inserts supplier_payments, supplier_payment_allocations, purchase_payments, and updates purchases.
  4. lib/financial.tsProvides allocateSupplierPaymentToOldestBills and supplier payable calculations.
  5. Tablessupplier_payments -> supplier_payment_allocations -> purchases -> purchase_payments.

Staff Salary Flow

Monthly ledger plus transaction history.

  1. app/staff/payment/create/page.tsxLoads staff options and existing salary transaction when editing.
  2. StaffSalaryPaymentFormCollects staff, month, salary settings, advance/salary type, amount, date, and note.
  3. app/actions.ts: upsertStaffSalaryPaymentUpserts monthly ledger, inserts/updates transaction, then syncs staff ledgers.
  4. lib/staff-payroll.tsRecalculates total advance, salary paid, remaining, carry forward, and status.
  5. app/staff/page.tsxDisplays staff summaries, ledgers, and transactions.

Dashboard Flow

The dashboard reads many modules and summarizes business state.

  1. app/page.tsxFetches sales, purchases, expenses, staff ledgers, customers, vendors, and activity logs.
  2. lib/financial.tsCalculates dashboard totals and payment states.
  3. lib/staff-payroll.tsBuilds staff salary snapshots used by dashboard/staff pages.
  4. components/dashboard/Renders KPI cards, charts, activity cards, report buttons, and tables.
  5. app/activity/page.tsxShows full activity records that the dashboard previews.

Technical Q&A

Common programming and architecture questions someone may ask about the system.

Which programming language is used?

The application is written in TypeScript. The frontend and backend route logic use Next.js with React components, server components, and server actions.

Which framework is used for the app?

The app uses Next.js App Router. Pages are inside app/, reusable UI is inside components/, and shared helper logic is inside lib/.

What is used for the backend?

Supabase is used as the backend platform. The app uses Supabase PostgreSQL for data, Supabase Auth for authentication, and Supabase server clients for protected database reads and writes.

Where is the data saved?

Business data is saved in Supabase PostgreSQL tables such as sales, purchases, customers, vendors, products, staff_salary_ledgers, and activity_logs.

Where is the database schema defined?

The full schema is kept in supabase/schema.sql. Smaller update scripts are kept in supabase/migrations/ so changes can be tracked and applied step by step.

How does the app connect to Supabase?

Server-side pages and actions use lib/supabase/server.ts. Browser-side code can use lib/supabase/client.ts when client access is needed.

Where is create, update, delete, and payment logic written?

Most business writes are centralized in app/actions.ts. Examples include upsertSale, upsertPurchase, createCustomerPayment, createSupplierPayment, and upsertStaffSalaryPayment.

How are financial calculations handled?

Shared calculations live in lib/financial.ts and staff salary calculations live in lib/staff-payroll.ts. This keeps payment status, remaining balance, supplier payable, and salary ledger logic consistent across pages.

Does the system use APIs?

Most writes use Next.js server actions instead of separate REST endpoints. Supabase itself provides the database API, and PostgreSQL RPC is used where a safer transaction is needed, such as customer payment allocation.

How are customer and supplier payments stored?

Customer payments are stored in customer_payments and allocated through customer_payment_allocations. Supplier payments are stored in supplier_payments and allocated through supplier_payment_allocations. Bill-level histories are also stored in sales_payments and purchase_payments.

How is transactional safety handled?

Some flows are handled in the application action layer, while the safer customer payment allocation flow uses a PostgreSQL RPC transaction. If one insert or update fails in that RPC, the whole payment allocation is rolled back.

How is the activity log created?

Server actions call recordActivity in app/actions.ts. Those records are saved in activity_logs and displayed on app/activity/page.tsx.

How does language switching work?

Translation messages are stored in messages/en.ts and messages/ne.ts. Locale helpers in lib/i18n.ts and lib/i18n-server.ts choose English or Nepali based on the selected language.

How are Nepali dates handled?

Nepali date conversion and display helpers are inside lib/nepali-date.ts. Forms can show BS dates while the app stores comparable AD date values for filtering and reporting.

What styling system is used?

The UI uses Tailwind-style utility classes inside React components. Shared layout pieces such as cards, headers, buttons, badges, tables, and report toolbars live in components/.

Is it built for one business or SaaS?

The current version is suitable for one organization and one owner/admin account. To make it multi-tenant SaaS later, the database would need organization ownership fields, stricter RLS policies, and tenant-aware queries.

Can the system be backed up?

The settings page already has backup UI placeholders for CSV and ZIP exports. Actual backup logic can be added later by reading Supabase tables and exporting sales, customers, purchases, staff salary, and reports.

Where should a developer start if they want to change a feature?

Start from the route in app/, then follow the component in components/, then the form action in app/actions.ts, then the helper logic in lib/, and finally the related table in supabase/schema.sql.
